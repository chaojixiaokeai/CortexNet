# CortexNet 路线图与版本策略

版本：3.2.1

---

## 1. 版本策略

采用语义化版本：`MAJOR.MINOR.PATCH`。

1. `MAJOR`：破坏性 API 变化。
2. `MINOR`：向后兼容的新功能。
3. `PATCH`：缺陷修复与文档/测试改进。

统一对外模型入口长期保持 `CortexNet`。

---

## 2. 已完成里程碑

1. 包结构规范化，可直接发布到 PyPI。
2. 主类命名统一为 `CortexNet`，历史命名转为兼容别名。
3. CI、CodeQL、Dependabot、发布工作流已接入。
4. 基准、评测、聊天与开发脚本按目录分类。
5. 基础文档、模块映射、贡献与安全文档已补齐。

---

## 3. 近期路线（1-2 个版本）

1. 完善双语文档深度与示例覆盖。
2. 增强基准结果结构化输出与可视化脚本。
3. 补充更多迁移样例与模型族兼容测试。
4. 提升 `from_pretrained` 错误提示可读性。

---

## 4. 中期路线（3-4 个版本）

1. 引入更系统的性能回归门禁（阈值化）。
2. 优化长上下文下缓存与注意力协同策略。
3. 增加训练模板脚本（单机 + 分布式标准模板）。
4. 增加更多设备后端验证矩阵。

---

## 5. 长期方向

1. 构建统一实验基线集合，支持社区可复现对比。
2. 建立可插拔能力模块协议，降低二次开发成本。
3. 形成稳定的插件生态（适配器、评测器、部署模板）。
4. 完善开源治理：贡献评审 SLA、版本支持策略、LTS 候选。

---

## 6. 质量门禁策略

每次发布至少满足：

1. `make lint` 通过。
2. `make test-all` 通过。
3. `make check` 通过。
4. `CHANGELOG.md` 已更新。

---

## 7. 向后兼容承诺

1. 主类 `CortexNet` 与核心配置 `CortexNetConfig` 保持稳定。
2. 旧别名在明确公告前不直接移除。
3. 移除行为需经历：弃用标记 -> 迁移文档 -> 大版本移除。

---

## 8. 社区协作计划

1. 增加标准 issue 模板与 PR 模板使用率。
2. 建立“good first issue”与“help wanted”标签机制。
3. 增加对基准报告、文档改进类贡献的明确激励。

---

## 9. 风险项

1. 多后端支持带来测试矩阵膨胀。
2. 适配器扩展过快可能导致映射规则维护成本上升。
3. 基准口径不统一会影响社区结论可信度。

---

## 10. 维护原则

1. 文档与代码同版本推进。
2. 优先可验证改动，减少“只改叙述不改测试”。
3. 对外承诺以仓库主分支和 Release 说明为准。
