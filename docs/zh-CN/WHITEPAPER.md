版本：3.2.1  
文档类型：技术白皮书（研究与工程并重）  
适用对象：研究者、工程团队、开源使用者、技术决策者

---

## 摘要

CortexNet 是一个统一的神经网络架构实现，目标是在保留 Transformer 表达能力的同时，降低长序列推理成本并提升可扩展性。系统将多尺度状态空间模块（SSM）、选择性稀疏注意力、突触记忆、条件路由（MoE）与可选进阶能力模块整合在同一模型框架内，并提供从主流开源模型到 CortexNet 的适配与迁移路径。

本文给出 CortexNet 的问题定义、设计目标、核心机制、复杂度分析、工程化策略、评测方案、已知边界和后续演进方向。

---

## 1. 问题定义

在大规模语言模型工程中，常见矛盾包括：

1. 长上下文下二次复杂度注意力带来的时延与显存压力。
2. 全参数激活路径导致推理成本高、吞吐受限。
3. 纯注意力机制对历史信息的“硬缓存”过重，灵活记忆能力不足。
4. 研究模块与工程模块耦合，难以在开源场景下长期维护。

CortexNet 的目标不是完全替代 Transformer，而是在统一接口下提供更可控的“混合路径”能力，支持按场景选择 Lite / Compatibility / Full 运行模式。

---

## 2. 设计目标

### 2.1 功能目标

1. 保持语言建模任务中的可用精度与泛化能力。
2. 提供可配置的长上下文效率优化路径。
3. 支持从主流 HuggingFace 模型目录进行自动识别、配置转换、权重映射。
4. 统一对外 API 为 `CortexNet`，降低集成成本。

### 2.2 工程目标

1. `pip` 可安装、`python -m cortexnet --smoke-test` 可验证。
2. 代码分层清晰，支持开源协作和 CI 自动校验。
3. 在不同设备（CPU/CUDA/MPS/NPU/MLU）下具备最小可运行路径。

---

## 3. 核心机制

### 3.1 多尺度状态空间模块（SSM）

SSM 路径承担线性复杂度序列建模职责。直观上可表示为：

- 状态更新：`h_t = A h_{t-1} + B x_t`
- 输出映射：`y_t = C h_t + D x_t`

多尺度结构通过并行状态子空间提升不同时间尺度的建模能力，并支持增量状态缓存，适配自回归解码。

### 3.2 选择性稀疏注意力

注意力路径保留关键 token 交互，同时通过 Top-K / 滑窗等策略降低平均计算量。理论上复杂度由 `O(n^2)` 降为 `O(n*k)`（`k << n`）。

### 3.3 突触记忆

记忆路径提供显式、可衰减的历史信息聚合通道，用于补足纯缓存机制在跨段上下文迁移时的灵活性。

### 3.4 条件路由与专家网络

MoE 仅激活部分专家，降低每 token 有效计算。系统通过容量控制和辅助均衡损失降低专家塌缩风险。

### 3.5 自适应融合门

融合门对不同路径输出进行动态加权，按输入分布和上下文阶段调整 SSM、Attention、Memory 的相对贡献。

## 4. 架构形态

### 4.1 训练/前向流程

`Token IDs -> Embedding -> N x CortexBlock -> Final Norm -> LM Head -> Logits`

每个 `CortexBlock` 内部包含：

- 规范化
- SSM 路径
- 稀疏注意力路径
- 记忆路径
- 融合
- FFN / MoE

### 4.2 推理流程

- 支持缓存式增量解码
- 支持 `temperature/top_k/top_p/repetition_penalty`
- 支持 Lite 模式下 SSM-only 解码阈值策略
- 支持 lazy 设备预热，降低冷启动体感

---

## 5. 复杂度与资源侧分析

### 5.1 理论复杂度（主要项）

1. 稠密注意力：`O(n^2 * d)`
2. 稀疏注意力：`O(n * k * d)`
3. SSM：`O(n * d)`

CortexNet 通过“SSM + 稀疏注意力”混合路径降低长上下文下的平均计算开销。

### 5.2 推理内存要点

1. KV Cache 受 `num_kv_heads`、序列长度、dtype 影响显著。
2. GQA 与稀疏策略可降低缓存压力。
3. Lazy 设备与映射缓存可优化冷启动时延。

---

## 6. 可迁移性与生态兼容

CortexNet 提供 `from_pretrained` 流程：

1. 识别模型类型（如 LLaMA/Qwen/Mistral/Baichuan/ChatGLM/Phi/Gemma 等）
2. 读取源配置并转换为 `CortexNetConfig`
3. 权重命名映射与形状适配
4. 架构适配与可选校准
5. 设备/精度策略落地

这使其可作为“统一运行接口层”，在不破坏原始模型资产管理流程的前提下完成迁移验证。

## 8. 应用场景

1. 本地推理与对比实验：快速对比 CortexNet 与 Transformer 路径。
2. 开源模型兼容运行：统一加载接口降低维护多套推理代码成本。
3. 长上下文实验：通过脚本对延迟、吞吐、内存进行量化评估。
4. 架构研究试验：在 Full 模式下开展图推理、因果推理、多智能体等方向探索。

---

## 9. 已知边界

1. 部分第三方依赖存在 deprecation warning，不影响主路径可用性。
2. 真实业务吞吐与时延必须在目标硬件和目标负载下复测。
3. 权重映射依赖源模型命名规范，极端自定义权重需补充映射规则。
4. Full 模式并非所有生产场景默认最优，应按目标成本选择 Lite/Compatibility。

---

## 10. 路线与治理建议

1. 坚持“统一主类 `CortexNet` + 兼容别名保留”的渐进迁移策略。
2. 新功能必须附带测试与文档，保持可维护性。
3. 每个发行版本同步更新 CHANGELOG、文档索引与发布流程。
4. 优先补齐基准数据集标准化与可复现实验配置。

---

## 结论

CortexNet 的核心价值在于：

1. 提供混合路径架构以平衡表达能力与效率。
2. 提供工程可落地的迁移、部署、测试和发布闭环。
3. 提供适合开源协作的模块化边界与质量门禁。

对希望同时兼顾研究探索与工程交付的团队，CortexNet 可作为统一的模型运行与演进基座。
