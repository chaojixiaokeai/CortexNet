# CortexNet 开源发布与协作指南

版本：3.2.1

---

## 1. 文档目标

帮助维护者与贡献者快速理解 CortexNet 的开源协作流程，包括：

1. 提交规范
2. 质量门禁
3. 发布流程
4. 安全响应

---

## 2. 仓库结构约定

1. `cortexnet/`：可发布库代码。
2. `scripts/`：运行脚本（基准、评测、聊天、开发工具）。
3. `tests/`：单测与回归测试。
4. `docs/`：双语文档。

---

## 3. 提交流程

建议标准流程：

1. 新建分支开发。
2. 本地执行 `make lint && make test-all && make check`。
3. 提交 PR，等待 CI 全绿。
4. Reviewer 完成技术审查后合并。

---

## 4. 代码规范

1. 默认 UTF-8、LF 换行。
2. Python 保持可读性优先，避免过度魔法逻辑。
3. 新模块需附带最小测试。
4. 对外 API 变更必须更新文档与 changelog。

---

## 5. 发布流程

### 5.1 版本准备

1. 更新 `pyproject.toml` 版本号。
2. 更新 `CHANGELOG.md`。
3. 确认测试和构建通过。

### 5.2 GitHub Release

发布 Release 后会触发 `publish.yml` 自动发布 PyPI（需 `PYPI_API_TOKEN`）。

---

## 6. Issue 与 PR 建议模板

Issue 至少包含：

1. 复现步骤
2. 环境信息
3. 期望行为与实际行为

PR 至少包含：

1. 改动摘要
2. 影响范围
3. 测试结果
4. 文档更新情况

---

## 7. 安全与合规

1. 漏洞反馈流程见 `SECURITY.md`。
2. 许可证为 Apache-2.0，需保留版权与许可声明。
3. 引入新依赖需评估许可证兼容性与安全风险。

---

## 8. 社区贡献建议

适合新贡献者的方向：

1. 文档补充与示例完善。
2. 新模型族映射规则补充。
3. 基准脚本结果导出改进。
4. 低风险 bugfix 与测试增强。

---

## 9. 维护者值守建议

1. 每周检查依赖更新与安全告警。
2. 定期归档和整理长期未响应 issue。
3. 在版本迭代前做一次回归基准巡检。

---

## 10. 成功标准

开源协作质量可用以下指标衡量：

1. PR 合并前 CI 通过率。
2. 新功能测试覆盖率。
3. 发布后回归缺陷数量。
4. 文档更新及时率。
